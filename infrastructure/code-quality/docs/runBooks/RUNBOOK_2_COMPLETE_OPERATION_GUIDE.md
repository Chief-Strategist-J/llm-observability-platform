# Runbook 2: Complete Operation Guide - How to Run and Configure Everything

## Overview
This runbook provides comprehensive instructions for running and configuring the SonarQube analysis system, including all commands, file explanations, and operational procedures.

## Prerequisites
- Completed Runbook 1 (Project Setup)
- SonarQube server running
- Project configured with `sonar-config.json`

## Quick Start Commands

### One-Command Analysis
```bash
# Run complete analysis (coverage + scan)
make sonar-analyze

# Or run orchestrator directly
bash scripts/sonar/orchestrator.sh
```

### Individual Commands
```bash
# Install dependencies
make sonar-setup

# Generate coverage only
make sonar-coverage

# Run scan only
make sonar-scan

# Start SonarQube server
make sonar-start

# Stop SonarQube server
make sonar-stop
```

## Detailed Command Reference

### 1. Server Management

#### Start SonarQube Server
```bash
# Method 1: Using Makefile
make sonar-start

# Method 2: Direct script
bash scripts/sonar/server_manager.sh start_server

# Method 3: Manual Docker Compose
cd ../code-quality/deployment/docker
docker-compose up -d db
sleep 30
docker-compose up -d sonarqube
```

**What happens:**
1. Starts PostgreSQL database service
2. Waits 30 seconds for database initialization
3. Starts SonarQube web service
4. Services connect via Docker network

#### Stop SonarQube Server
```bash
# Method 1: Using Makefile
make sonar-stop

# Method 2: Direct script
bash scripts/sonar/server_manager.sh stop_server

# Method 3: Manual Docker Compose
cd ../code-quality/deployment/docker
docker-compose down
```

#### Check Server Status
```bash
# Check if server is running
docker-compose ps | grep sonarqube

# Check server health
curl "http://localhost:9000/api/system/status"

# Expected response: {"status":"UP"}
```

### 2. Dependency Management

#### Install Analysis Dependencies
```bash
# Method 1: Using Makefile
```

**What gets installed:**


### 3. Coverage Generation

#### Generate Coverage Report
```bash
# Method 1: Using Makefile
make sonar-coverage

# Method 2: Direct script
bash scripts/sonar/coverage_generator.sh

# Method 3: Manual command
cargo tarpaulin --output-dir coverage --skip-clean
```

**What happens:**
1. Runs all project tests
2. Instruments code during execution
3. Generates coverage data
4. Creates `coverage/cobertura.xml` file

#### Coverage Report Location
```bash
# Check coverage report
ls -la coverage/

# View coverage summary
cat coverage/cobertura.xml | head -5

# Expected: XML coverage data with line counts
```

### 4. SonarQube Scanning

#### Run Complete Analysis
```bash

bash scripts/sonar/orchestrator.sh
```

```bash

# Method 2: Direct script

  -e SONAR_HOST_URL="http://sonarqube:9000" \
  -e SONAR_TOKEN="squ_44characterlongtoken..." \
  -v "$(pwd):/project" \
  sonarsource/sonar-scanner-cli \
  -Dproject.settings=/project/sonar-project.properties \
  -Dsonar.projectBaseDir=/project
```

## File Structure and Explanations

### Core Configuration Files

#### `sonar-config.json`
**Purpose**: Single source of truth for project configuration
**Location**: Project root directory

```json
{
  "projectKey": "distributed-trace-analysis-engine",
  "projectName": "Distributed Trace Analysis Engine", 
  "projectVersion": "0.1.0",
  "language": "rust",
  "sourceDirs": ["src"],
  "testDirs": ["tests"],
  "exclusions": ["target/**", "web/**", "*.json"],
  "sonarHostUrl": "http://sonarqube:9000",
  "sonarToken": "squ_44characterlongtoken...",
  "coverage": {
    "tool": "cargo-tarpaulin",
    "outputFormat": "xml", 
    "outputDir": "coverage"
  }
}
```

**Key Fields**:
- `projectKey`: Unique identifier (no spaces)
- `sonarToken`: 44-character authentication token
- `sonarHostUrl`: Docker network URL for scanner

#### `sonar-project.properties`
**Purpose**: Generated file for SonarQube scanner
**Location**: Project root directory
**Generated by**: `scripts/sonar/sonar_scanner.sh`

```properties
sonar.projectKey=distributed-trace-analysis-engine
sonar.projectName=Distributed Trace Analysis Engine
sonar.projectVersion=0.1.0
sonar.language=rust
sonar.sources=src
sonar.exclusions=target/**,web/**,node_modules/**,*.json,*.yaml,*.yml
sonar.host.url=http://sonarqube:9000
sonar.token=squ_44characterlongtoken...
```

### Script Files

#### `scripts/sonar/utils.sh`
**Purpose**: Common utility functions
**Functions**: `log_info()`, `log_error()`, `log_success()`, `check_command()`

#### `scripts/sonar/config_loader.sh`
**Purpose**: Load and validate configuration
**Functions**: `load_config()` - reads JSON and exports environment variables

#### `scripts/sonar/server_manager.sh`
**Purpose**: Manage SonarQube server lifecycle
**Functions**: `start_server()`, `stop_server()`

#### `scripts/sonar/coverage_generator.sh`
**Purpose**: Generate code coverage reports
**Functions**: `install_tarpaulin()`, `generate_coverage()`

#### `scripts/sonar/sonar_scanner.sh`
**Purpose**: Run SonarQube scanner
**Functions**: `create_properties()`, `run_scan()`

#### `scripts/sonar/orchestrator.sh`
**Purpose**: Coordinate complete analysis workflow
**Functions**: Runs coverage → scanner in sequence

### Build Files

#### `Makefile`
**Purpose**: Convenient command interface
**Targets**: `sonar-analyze`, `sonar-setup`, `sonar-coverage`, `sonar-start`, `sonar-stop`, `sonar-scan`

```makefile
.PHONY: sonar-analyze sonar-setup sonar-coverage sonar-start sonar-stop sonar-scan

SCRIPTS_DIR := scripts/sonar

sonar-analyze:
	@bash $(SCRIPTS_DIR)/orchestrator.sh

sonar-setup:
	@cargo install cargo-tarpaulin
	@echo "Dependencies installed successfully"

sonar-coverage:
	@bash $(SCRIPTS_DIR)/coverage_generator.sh

sonar-start:
	@bash $(SCRIPTS_DIR)/server_manager.sh start_server

sonar-stop:
	@bash $(SCRIPTS_DIR)/server_manager.sh stop_server

sonar-scan:
	@bash $(SCRIPTS_DIR)/sonar_scanner.sh
```

## Configuration Management

### Environment Variables
The system uses these environment variables (loaded from `sonar-config.json`):

```bash
SONAR_PROJECT_KEY=distributed-trace-analysis-engine
SONAR_PROJECT_NAME=Distributed Trace Analysis Engine
SONAR_PROJECT_VERSION=0.1.0
SONAR_LANGUAGE=rust
SONAR_SOURCE_DIRS=src
SONAR_TEST_DIRS=tests
SONAR_EXCLUSIONS=target/**,web/**,node_modules/**,*.json,*.yaml,*.yml
SONAR_HOST_URL=http://sonarqube:9000
SONAR_TOKEN=squ_44characterlongtoken...
COVERAGE_TOOL=cargo-tarpaulin
COVERAGE_OUTPUT_FORMAT=xml
COVERAGE_OUTPUT_DIR=coverage
```

### Customizing Configuration

#### Change Project Details
```bash
# Edit sonar-config.json
vim sonar-config.json

# Update fields:
# - projectName: Display name
# - projectVersion: Current version
# - sourceDirs: Source code locations
# - exclusions: Files to ignore
```

#### Update Token
```bash
# Generate new token
curl -X POST -u admin:admin \
  "http://localhost:9000/api/user_tokens/generate?name=new-token"

# Update sonar-config.json with new token
# Then regenerate properties
bash scripts/sonar/sonar_scanner.sh create_properties
```

#### Modify Coverage Settings
```bash
# Edit coverage section in sonar-config.json
"coverage": {
  "tool": "cargo-tarpaulin",
  "outputFormat": "xml",
  "outputDir": "coverage",
  "additionalArgs": ["--skip-clean", "--ignore-tests"]
}
```

### CI/CD Integration Workflow
```bash

# 2. Wait for server readiness

# 3. Install dependencies

# 4. Run complete analysis
make sonar-analyze

# 5. Check results via API
curl -u "$SONAR_TOKEN:" \
  "http://localhost:9000/api/measures/component?component=$PROJECT_KEY&metricKeys=coverage,violations"
```

## Best Practices

### Development Best Practices
1. **Run analysis regularly**: After significant code changes
2. **Check results promptly**: Fix issues before they accumulate
3. **Monitor coverage**: Aim for >80% coverage on critical code
4. **Review violations**: Address high-priority issues first

### Configuration Best Practices
1. **Use version control**: Commit `sonar-config.json`
2. **Document changes**: Update team on configuration updates
3. **Test tokens**: Validate new tokens before deployment
4. **Backup configuration**: Keep copy of working config

### Security Best Practices
1. **Rotate tokens**: Every 90 days
2. **Limit permissions**: Use project-specific tokens
3. **Monitor access**: Review token usage regularly
4. **Secure storage**: Don't commit tokens to repositories
